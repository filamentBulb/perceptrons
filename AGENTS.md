# Repository Guidelines

## Project Structure & Module Organization

This is a TanStack Start React app using Vite, TypeScript, Tailwind CSS, and pnpm. Application code lives in `src/`. File-based routes are under `src/routes/`; shared UI is in `src/components/`, with reusable shadcn-style primitives in `src/components/ui/`. Shared hooks live in `src/hooks/`, utilities and stores in `src/lib/` and `src/utils/`, and static/data fixtures in `src/data/`. Public browser assets belong in `public/`. Do not manually edit `src/routeTree.gen.ts`; it is generated by TanStack Router.

## Build, Test, and Development Commands

- `pnpm install`: install dependencies from `pnpm-lock.yaml`.
- `pnpm dev`: start the local Vite dev server on port `3000`.
- `pnpm build`: create a production build, including the Nitro server output.
- `pnpm preview`: preview the built app locally.
- `pnpm test`: run the Vitest suite once.
- `pnpm lint`: run Biome lint checks.
- `pnpm format`: format files with Biome.
- `pnpm check`: run Biome formatting and lint validation together.

## Coding Style & Naming Conventions

Use TypeScript and React function components. Follow existing route, component, and hook patterns before adding new abstractions. Biome is the source of truth for formatting: tabs for indentation and double quotes for JavaScript/TypeScript strings. Prefer the configured `#/*` import alias for source imports when it improves clarity. Use `PascalCase` for components, `camelCase` for functions and variables, and `useName` for hooks. Route files should follow TanStack Router conventions, such as `about.tsx`, `demo/table.tsx`, or `$guitarId.tsx`.

## Testing Guidelines

Vitest is the configured test runner, with Testing Library available for React tests. Add focused tests for new user-facing behavior, stores, utilities, and bug fixes. Name tests with `.test.ts` or `.test.tsx` near the code they cover, or place related suites in a local `__tests__/` directory. Run `pnpm test` before handing off substantive changes.

## Commit & Pull Request Guidelines

The current history uses short, imperative commit messages such as `build fix + ui updates` and `added skills`. Keep commits concise and scoped to one logical change. For pull requests, include a brief summary, the verification commands run, linked issues when applicable, and screenshots or recordings for UI changes. Note any required environment variables, such as AI provider API keys, without committing secrets.

## Security & Configuration Tips

Keep API keys and local secrets in ignored environment files. Review server routes and AI-related handlers carefully before logging request data or provider responses.
